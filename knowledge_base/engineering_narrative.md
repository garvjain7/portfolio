# Engineering Narrative

## Entry Point

Garv did not start with system design or backend architecture. The entry point was basic web development — HTML, CSS, JavaScript — building simple things and seeing visible results. What held his attention early was not the technology itself but the ability to create something from scratch and make it available for others to use. That pull toward building rather than consuming technology has remained consistent.

## When Projects Replaced Tutorials

The shift happened when projects stopped fitting inside tutorial boundaries. Once a project had real requirements, tutorials stopped providing complete answers. The question changed from "how do I use this technology" to "how do I make this work for my specific situation." That forced him to combine concepts, make design decisions without step-by-step guidance, and debug problems that had no direct precedent in anything he had studied.

Projects exposed gaps that structured courses never covered. That made building more interesting than studying technologies in isolation — not because courses were without value, but because real constraints produced questions that only building could answer.

## The Shift From Features to Systems

Early on, the goal was working features. If a feature worked, that was sufficient.

Over time a different pattern emerged: the feature itself was usually the easier part. The harder questions were about how data moved through the system, how components interacted, how information was stored and retrieved, and why certain architectural decisions existed.

ApnaEV was one early example of this. What started as an EV station discovery platform became an exploration of how mapping systems represent road networks, how routing engines model geography, and how location-aware applications are structured internally. The visible product was simple. The questions it generated were not.

This pattern repeated across projects. The more Garv built, the more his attention moved from the surface behavior of a system to the mechanisms producing it. That transition — from feature-focused to systems-focused — did not happen at a single moment. It accumulated across multiple projects as the interesting problems consistently turned out to be structural rather than functional.

## How Backend, Data, and AI Converged

These three areas did not develop as separate interests that later connected. They developed sequentially, each one pulling the next into view.

Backend came first because every real project needed APIs, authentication, databases, and business logic. Building backend systems naturally produced questions about how information should be structured, stored, and retrieved — which expanded into database design and data systems more broadly.

AI entered later, through the realization that many of the most interesting problems in software involved making systems intelligent or data-driven. Once Garv started working with LLM integrations, retrieval systems, and AI pipelines, the relationship between the three areas became structurally clear: backend systems manage information, data systems organize it, AI systems extract value from it. Three layers of the same problem rather than three separate fields.

This is why his projects consistently combine all three — ScanVista, DataInsights.ai, ComplySense, and the portfolio RAG backend each involve backend infrastructure, a data layer, and an AI component working together.

## How the Learning Process Developed

Most of Garv's engineering depth was built by starting projects slightly beyond his current knowledge and learning the missing pieces as obstacles appeared. The approach produced mistakes, redesigns, and dead ends — and also exposed him to a wider range of engineering problems than a sequential curriculum would have.

The concepts he understands most deeply today were almost all learned because a project demanded them. Operational Transformation for CodeAlive's collaboration engine. Multi-tenant schema isolation for DataInsights.ai. Chunk boundary design for the portfolio RAG system. In each case the learning was driven by a concrete problem, not a syllabus.

The cumulative effect of this approach is a shift in how software looks. It no longer appears as a collection of technologies to be learned. It appears as a collection of interacting systems, constraints, and tradeoffs — each design encoding decisions made under specific conditions, each architecture reflecting the problems it was built to solve.
